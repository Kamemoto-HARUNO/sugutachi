# 画面一覧・ユーザーフロー定義

## 1. 目的

このドキュメントは、MVPで必要な画面、画面遷移、予約ステータス、主要な入力項目を整理し、次工程のDB設計・API設計・UI実装へ接続するためのもの。

MVPでは、リラクゼーション、ボディケア、もみほぐしのマッチングに必要な画面に限定する。STI検査結果、LINE連携、高度なリアルタイム位置表示、自動出金・即時出金は対象外とする。

## 2. ロール方針

### 2.1 ロール
* **ゲスト**: 未ログインユーザー。
* **ユーザー**: リラクゼーション、ボディケア、もみほぐしを受ける側。
* **セラピスト**: リラクゼーション、ボディケア、もみほぐしを提供する側。
* **運営管理者**: 審査、通報、返金、出金、コンテンツ管理を行う。

### 2.2 アカウント方針
* 1つのアカウントがユーザー・セラピスト両方のロールを持てる設計にする。
* MVPの画面上は、ログイン後に「利用する」「提供する」を切り替える。
* 本人確認、年齢確認、電話番号確認はアカウント単位で保持する。
* セラピストとして稼働する場合は、追加でセラピスト審査、Stripe Connect登録、提供プロフィール登録を必須とする。

### 2.3 フロントエンドURL方針
* 公開ページと共通ページはプレフィックスなしで配置する。
* 利用者向け画面は `/user/*`、セラピスト向け画面は `/therapist/*`、運営管理画面は `/admin/*` に統一する。
* 詳細画面のパラメータは、原則としてAPIと同じ `public_id` を使う。
* 予約作成のような多段階フローは、1画面内ステップだけで閉じず、再読込に耐えられるようURLを分ける。
* キャンセル、返金申請、通報、レビュー投稿のような補助導線は、MVPでは専用サブパスを持たせる。UI上はモーダル表示でもよいが、URL遷移可能な構造を前提にする。
* ログイン後の初期遷移は、複数ロールを持つ場合は `/role-select`、利用ロールのみの場合は該当ホームへ遷移する。
* セラピストロールを持つが提供準備が未完了の場合は、`/therapist/onboarding` から必要画面へ案内する。

| 監査ログ | 管理操作の追跡 | 個人情報閲覧、更新、CSV出力、凍結、返金 | Yes |

## 7. 主要ユーザーフロー

### 7.1 ユーザー初回登録フロー
1. 公開トップを開く。
2. 新規登録する。
3. 利用規約、プライバシーポリシー、18歳以上、リラクゼーション目的に同意する。
4. 電話番号またはメール認証を行う。
5. 本人確認と年齢確認を行う。
6. 男性専用サービスの利用対象であることを自己申告する。
7. ユーザープロフィールを入力する。
8. 任意で性的指向・性自認等を入力し、セラピストへ表示する範囲を選ぶ。
9. 通知許可を設定する。
10. ユーザーホームへ遷移する。

### 7.2 セラピスト初回登録フロー
1. 新規登録または既存アカウントでログインする。
2. ロール選択で「提供する」を選ぶ。
3. セラピスト向け取引条件、禁止事項、直接取引禁止、医療・治療目的ではないことに同意する。
4. 本人確認と年齢確認を完了する。
5. Stripe Connect登録を開始し、Stripe-hosted onboardingで本人確認・銀行口座登録を行う。
6. 提供プロフィール、写真、自己紹介、提供メニュー、対応エリアを登録する。
7. 料金設定を登録する。
8. 運営のプロフィール・写真・研修ステータス審査を受ける。
9. 承認後、稼働管理でオンラインにできる。

### 7.3 公開プロフィール閲覧フロー
1. ゲストまたはユーザーが、公開トップまたはSNS等の外部導線からセラピスト詳細を開く。
2. 写真、自己紹介、提供メニュー、公開レビュー、セラピスト都合キャンセル回数を確認する。
3. 空き時間確認、見積もり、予約リクエストへ進みたい場合はログインまたは新規登録へ進む。

### 7.4 オンデマンド予約フロー
1. ユーザーが施術場所を入力する。
2. ユーザーホームで近隣セラピストを検索する。
3. サーバー側で直線距離から徒歩目安レンジを概算し、一覧に表示する。
4. ユーザーがセラピスト詳細を確認する。
5. 予約リクエスト作成画面で、「今すぐ依頼」を選び、提供メニュー、施術時間、場所、体調確認を入力する。
6. 自動見積もりを行い、総額、手数料、キャンセル規定、禁止事項への同意を表示する。
7. ユーザーがクレジットカードで与信枠を確保する。
8. セラピストへリクエスト通知を送る。
9. セラピストが取引条件を確認して承諾または拒否する。
10. 承諾された場合、予約確定となる。
11. 拒否またはタイムアウトの場合、与信を解放し、ユーザーへ再検索を促す。

### 7.5 予定予約フロー
1. セラピストが空きスケジュール管理画面で、単発の空き時間、受付締切、予定予約用基本地点、必要に応じて枠ごとの出動拠点と公開エリア名を設定する。
2. ユーザーが施術場所を選択し、セラピスト一覧または詳細から予定予約導線へ進む。
3. ユーザーが空き時間選択画面で、日付ごとの連続予約可能時間帯と公開エリア名を確認する。
4. ユーザーが希望メニューと開始時刻を選ぶ。
5. システムが、受付締切、連続空き時間、枠ごとの出動拠点ベースの対応可否、同時保留件数上限、同一セラピストへの重複リクエスト禁止を確認する。
6. 自動見積もりを行い、総額、手数料、キャンセル規定、禁止事項への同意を表示する。
7. ユーザーがクレジットカードで与信枠を確保する。
8. 予定予約リクエストを作成し、対象時間帯を仮押さえする。
9. セラピストは、ユーザー概要、場所種別、概算移動負荷、受取予定額を確認し、承諾時に前後バッファ分数を入力する。
10. セラピストが承諾した場合、予約確定となる。
11. セラピストが拒否した場合、または仮押さえ期限を迎えた場合は、与信を解放し、空き時間を再公開する。

### 7.6 出張リクエストフロー
1. ユーザーがセラピスト詳細から、通常予約できない場合の補助導線として出張リクエスト送信へ進む。
2. 希望エリアとして都道府県を選択し、任意メッセージを入力する。
3. システムが、相互ブロック、短期間の重複送信、アカウント単位の送信上限、連絡先交換表現の有無を確認する。
4. 問題がなければ、セラピストへ出張リクエストを保存し、アプリ内通知を送る。
5. セラピストはマイページの出張リクエスト一覧で、希望都道府県、メッセージ、受信日時を確認する。
6. MVPでは、セラピストからアプリ内で直接返信は行わず、今後の出動検討や空き枠公開判断の参考情報として扱う。

### 7.7 施術当日フロー
1. セラピストが移動開始を押す。
2. セラピストが到着を押す。
3. 必要に応じてアプリ内メッセージで場所確認を行う。
4. ユーザーとセラピストが、リラクゼーション目的、体調、禁止事項に同意する。
5. セラピストが施術開始を押す。
6. セラピストが施術終了を押す。
7. ユーザーが施術終了を確認する。
8. 決済を確定する。
9. 双方がレビューを投稿する。

### 7.8 キャンセルフロー
1. ユーザーまたはセラピストが予約詳細からキャンセルを選ぶ。
2. キャンセル理由を選ぶ。セラピスト都合キャンセルでは、ユーザー向け説明文の入力を必須とする。
3. システムがキャンセル時刻と規定に基づきキャンセル料を計算する。
4. ユーザーに請求額または返金予定を表示する。
5. キャンセルを確定する。
6. 必要に応じてカード与信の解放、部分決済、返金を行う。
7. セラピスト都合キャンセルの場合、運営管理画面に品質確認対象として記録し、公開上のキャンセル回数にも反映する。

### 7.9 中断・通報フロー
1. 予約詳細から「施術を中断する」または「通報する」を選ぶ。
2. 中断/通報カテゴリを選択する。
3. 詳細内容を入力する。
4. 身の危険や体調急変がある場合は、警察・救急への連絡を優先する案内を表示する。
5. 対象予約を紛争対応ステータスにする。
6. 必要に応じて相手への通知、メッセージ停止、出金保留、予約制限を行う。
7. 運営が当日または翌営業日に確認する。

### 7.10 返金・紛争フロー
1. ユーザーが予約詳細から返金申請する。
2. 返金理由、詳細、証跡を入力する。
3. 対象予約のセラピスト売上を保留する。
4. 運営が予約履歴、メッセージ、チェックイン履歴、通報内容を確認する。
5. 全額返金、一部返金、返金不可を判断する。
6. Stripe Connect上で返金処理を行う。
7. 結果を双方へ通知する。

### 7.11 セラピスト出金フロー
1. セラピストが売上残高画面を開く。
2. 確定済み売上、保留中売上、出金申請中、支払済みを確認する。
3. 出金可能額が最低出金額以上であれば出金申請する。
4. システムが本人確認、Stripe Connected Account状態、銀行口座、規約違反調査中でないことを確認する。
5. 申請日から処理日を決定する。
6. 運営が5日、15日、25日に出金処理を行う。
7. Stripeおよび金融機関の処理結果を出金履歴へ反映する。

## 8. 予約ステータス定義

| ステータス | 意味 | 主な遷移元 | 主な遷移先 |
| --- | --- | --- | --- |
| draft | 予約作成中 | なし | quote_confirmed |
| quote_confirmed | 見積もり確認済み | draft | payment_authorized |
| payment_authorized | カード与信確保済み | quote_confirmed | requested, canceled_by_user |
| requested | セラピスト承諾待ち（予定予約では仮押さえ中） | payment_authorized | accepted, rejected, expired, canceled_by_user |
| accepted | セラピスト承諾済み（予定予約では前後バッファ入力済み） | requested | confirmed |
| confirmed | 予約確定 | accepted | moving, canceled_by_user, canceled_by_therapist |
| moving | セラピスト移動開始 | confirmed | arrived, canceled_by_therapist, interrupted |
| arrived | セラピスト到着 | moving | in_progress, interrupted |
| in_progress | 施術中 | arrived | completed, interrupted |
| completed | 施術終了 | in_progress | payment_captured, disputed |
| payment_captured | 決済確定 | completed | review_pending, disputed |
| review_pending | レビュー待ち | payment_captured | closed |
| closed | 通常完了 | review_pending | なし |
| rejected | セラピスト拒否 | requested | なし |
| expired | 承諾タイムアウト | requested | なし |
| canceled_by_user | ユーザーキャンセル | payment_authorized, requested, confirmed | refunded, closed |
| canceled_by_therapist | セラピストキャンセル | confirmed, moving | refunded, closed |
| interrupted | 中断 | moving, arrived, in_progress | disputed, payment_captured, refunded |
| disputed | 紛争対応中 | completed, payment_captured, interrupted | refunded, closed |
| refunded | 返金済み | canceled_by_user, canceled_by_therapist, disputed | closed |

## 11. MVP外にする画面

| 画面/機能 | 理由 |
| --- | --- |
| STI検査結果アップロード | 機微情報が重く、MVPでは不要 |
| LINE連携設定 | Web Pushで代替する |
| 正確な地図ピン表示 | 位置推定・待ち伏せリスクが高い |
| リアルタイム緊急ボタン | 運営の即時対応体制がないため |
| 自動出金・即時出金 | 運用確認後に検討 |
| 複数人同時予約 | 安全運用が複雑になるため |
| サブスクリプション | MVPの検証目的から外れる |

## 12. 次工程へのメモ

DB設計では、少なくとも以下の領域を分ける。

* accounts / users / therapist_profiles / user_profiles。
* identity_verifications。
* therapist_menus / pricing_rules。
* locations / service_addresses。
* bookings / booking_status_logs。
* payment_intents / stripe_connected_accounts / payouts。
* messages。
* reviews。
* reports / disputes / refunds。
* admin_audit_logs。

予約ステータス、出金ステータス、本人確認ステータス、セラピスト審査ステータスは、DB・API・画面で同じ名称を使う。
