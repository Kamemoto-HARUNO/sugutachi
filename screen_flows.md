# 画面一覧・ユーザーフロー定義

## 1. 目的

このドキュメントは、MVPで必要な画面、画面遷移、予約ステータス、主要な入力項目を整理し、次工程のDB設計・API設計・UI実装へ接続するためのもの。

MVPでは、リラクゼーション、ボディケア、もみほぐしのマッチングに必要な画面に限定する。STI検査結果、LINE連携、高度なリアルタイム位置表示、自動出金・即時出金は対象外とする。

## 2. ロール方針

### 2.1 ロール
* **ゲスト**: 未ログインユーザー。
* **ユーザー**: リラクゼーション、ボディケア、もみほぐしを受ける側。
* **タチキャスト**: リラクゼーション、ボディケア、もみほぐしを提供する側。
* **運営管理者**: 審査、通報、返金、出金、コンテンツ管理を行う。

### 2.2 アカウント方針
* 1つのアカウントがユーザー・タチキャスト両方のロールを持てる設計にする。
* 新規登録時は最初に使うロールを1つ選び、ログイン後に `/role-select` から不足ロールを追加できるようにする。
* MVPの画面上は、ログイン後に「利用する」「提供する」を切り替える。
* 本人確認、年齢確認、電話番号確認はアカウント単位で保持する。
* タチキャストとして稼働する場合は、追加でタチキャスト審査、受取口座登録、提供プロフィール登録を必須とする。

### 2.3 フロントエンドURL方針
* 公開ページと共通ページはプレフィックスなしで配置する。
* 利用者向け画面は `/user/*`、タチキャスト向け画面は `/therapist/*`、運営管理画面は `/admin/*` に統一する。
* 詳細画面のパラメータは、原則としてAPIと同じ `public_id` を使う。
* 予約作成のような多段階フローは、1画面内ステップだけで閉じず、再読込に耐えられるようURLを分ける。
* キャンセル、返金申請、通報、レビュー投稿のような補助導線は、MVPでは専用サブパスを持たせる。UI上はモーダル表示でもよいが、URL遷移可能な構造を前提にする。
* ログイン後の初期遷移は、複数ロールを持つ場合は `/role-select`、利用ロールのみの場合は該当ホームへ遷移する。
* タチキャストロールを持つが提供準備が未完了の場合は、`/therapist/onboarding` から必要画面へ案内する。

## 7. 主要ユーザーフロー

### 7.1 ユーザー初回登録フロー
1. 公開トップまたは「はじめての方へ」を開く。
2. 新規登録する。利用者向け登録キャンペーンがある場合はオファー内容を確認できる。
3. 利用規約、プライバシーポリシー、18歳以上、リラクゼーション目的に同意する。
4. 電話番号またはメール認証を行う。
5. 本人確認と年齢確認を行う。
6. 男性専用サービスの利用対象であることを自己申告する。
7. ユーザープロフィールを入力する。
8. 任意で性的指向・性自認等を入力し、タチキャストへ表示する範囲を選ぶ。
9. 通知許可を設定する。
10. ユーザーホームへ遷移する。
11. 初回予約割引キャンペーン対象の場合は、本人確認承認後に保有オファーが付与され、ダッシュボードのオファー一覧と最初の見積もり画面で有効期限と割引内容を確認できる。

### 7.2 タチキャスト初回登録フロー
1. 新規登録または既存アカウントでログインする。
2. ロール選択で「提供する」を選ぶ。
3. タチキャスト向け登録キャンペーンがある場合は、登録画面で特典内容を確認する。
4. タチキャスト向け取引条件、禁止事項、直接取引禁止、医療・治療目的ではないことに同意する。
5. 本人確認と年齢確認を完了する。
6. 受取口座設定を開き、銀行名、支店名、口座種別、口座番号、口座名義を登録する。
7. 提供プロフィール、写真、自己紹介、提供メニュー、対応エリアを登録する。
8. 料金設定を登録する。
9. 運営のプロフィール・写真・研修ステータス審査を受ける。
10. 承認後、登録特典キャンペーン対象なら売上残高へ特典が反映される。
11. 稼働管理でオンラインにできる。

### 7.3 公開プロフィール閲覧フロー
1. ゲストまたはユーザーが、公開トップ、「はじめての方へ」、またはSNS等の外部導線からタチキャスト詳細を開く。
2. 写真、自己紹介、提供メニュー、公開レビュー、タチキャスト都合キャンセル回数を確認する。
3. 期間限定の利用者向け予約キャンペーンが有効なら、予約CTA付近でオファー内容を確認する。
4. 空き時間確認、見積もり、予約リクエストへ進みたい場合はログインまたは新規登録へ進む。

### 7.4 オンデマンド予約フロー
1. ユーザーが施術場所を入力する。
2. ユーザーホームで近隣タチキャストを検索する。
3. サーバー側で直線距離から移動時間目安レンジを概算し、一覧に表示する。
4. ユーザーがタチキャスト詳細を確認する。
5. 利用者向け予約キャンペーンが有効なら、CTA付近のオファー内容を確認する。
6. 予約リクエスト作成画面で、「今すぐ依頼」を選び、提供メニュー、施術時間、場所、体調確認を入力する。
7. 自動見積もりを行い、総額、手数料、キャンセル規定、禁止事項への同意を表示する。割引対象なら割引内容とオファー有効期限も表示する。
8. ユーザーがクレジットカードで与信枠を確保する。
9. タチキャストへリクエスト通知を送る。
10. タチキャストが取引条件を確認して承諾または拒否する。
11. 承諾された場合、予約確定となる。初回予約オファーを確保していた場合はこの時点で消費し、失効・辞退・キャンセルになった場合は自動復活する。予約成立特典キャンペーン対象のタチキャストには売上残高へ特典を付与する。
12. 拒否またはタイムアウトの場合、与信を解放し、ユーザーへ再検索を促す。

### 7.5 予定予約フロー
1. タチキャストが空きスケジュール管理画面で、単発の空き時間、受付締切、予定予約用基本地点、必要に応じて枠ごとの出動拠点と公開エリア名を設定する。
2. ユーザーが施術場所を選択し、タチキャスト一覧または詳細から予定予約導線へ進む。
3. ユーザーが空き時間選択画面で、施術場所・メニュー・日付を指定し、日付ごとの連続予約可能時間帯から15分単位の開始候補と公開エリア名を確認する。
4. ユーザーが希望メニューと開始時刻を選ぶ。
5. システムが、受付締切、連続空き時間、枠ごとの出動拠点ベースの対応可否、同時保留件数上限、同一タチキャストへの重複リクエスト禁止を確認する。
6. 自動見積もりを行い、総額、手数料、キャンセル規定、禁止事項への同意を表示する。割引対象なら割引内容とオファー有効期限も表示する。
7. ユーザーがクレジットカードで与信枠を確保する。
8. 予定予約リクエストを作成し、対象時間帯を仮押さえする。
9. タチキャストは、ユーザー概要、場所種別、概算移動負荷、受取予定額を確認し、承諾時に前後バッファ分数を入力する。
10. タチキャストが承諾した場合、予約確定となる。初回予約オファーを確保していた場合はこの時点で消費し、失効・辞退・キャンセルになった場合は自動復活する。予約成立特典キャンペーン対象のタチキャストには売上残高へ特典を付与する。
11. タチキャストが拒否した場合、または仮押さえ期限を迎えた場合は、与信を解放し、空き時間を再公開する。

### 7.6 出張リクエストフロー
1. ユーザーがタチキャスト詳細から、通常予約できない場合の補助導線として出張リクエスト送信へ進む。
2. 希望エリアとして都道府県を選択し、任意メッセージを入力する。
3. システムが、相互ブロック、短期間の重複送信、アカウント単位の送信上限、連絡先交換表現の有無を確認する。
4. 問題がなければ、タチキャストへ出張リクエストを保存し、アプリ内通知を送る。
5. タチキャストはマイページの出張リクエスト一覧で、希望都道府県、メッセージ、受信日時を確認する。
6. MVPでは、タチキャストからアプリ内で直接返信は行わず、今後の出動検討や空き枠公開判断の参考情報として扱う。

### 7.7 施術当日フロー
1. タチキャストが移動開始を押す。
2. タチキャストが到着を押す。
3. 必要に応じてアプリ内メッセージで場所確認を行う。
4. ユーザーとタチキャストが、リラクゼーション目的、体調、禁止事項に同意する。
5. タチキャストが施術開始を押す。
6. タチキャストが施術終了を押す。
7. ユーザーが施術終了を確認する。
8. 決済を確定する。
9. 双方がレビューを投稿する。

### 7.8 キャンセルフロー
1. ユーザーまたはタチキャストが予約詳細からキャンセルを選ぶ。
2. キャンセル理由を選ぶ。タチキャスト都合キャンセルでは、ユーザー向け説明文の入力を必須とする。
3. システムがキャンセル時刻と規定に基づきキャンセル料を計算する。
4. ユーザーに請求額または返金予定を表示する。
5. キャンセルを確定する。
6. 必要に応じてカード与信の解放、部分決済、返金を行う。
7. タチキャスト都合キャンセルの場合、運営管理画面に品質確認対象として記録し、公開上のキャンセル回数にも反映する。

### 7.9 中断・通報フロー
1. 予約詳細から「施術を中断する」または「通報する」を選ぶ。
2. 中断/通報カテゴリを選択する。
3. 詳細内容を入力する。
4. 身の危険や体調急変がある場合は、警察・救急への連絡を優先する案内を表示する。
5. 対象予約を紛争対応ステータスにする。
6. 必要に応じて相手への通知、メッセージ停止、出金保留、予約制限を行う。
7. 運営が当日または翌営業日に確認する。

### 7.10 返金・紛争フロー
1. ユーザーが予約詳細から返金申請する。
2. 返金理由、詳細、証跡を入力する。
3. 対象予約のタチキャスト売上を保留する。
4. 運営が予約履歴、メッセージ、チェックイン履歴、通報内容を確認する。
5. 全額返金、一部返金、返金不可を判断する。
6. プラットフォームのStripe決済に対して返金処理を行う。
7. 結果を双方へ通知する。

### 7.11 タチキャスト出金フロー
1. タチキャストが売上残高画面を開く。
2. 確定済み売上、保留中売上、出金申請中、支払済みを確認する。
3. 出金可能額が最低出金額以上であれば出金申請する。
4. システムが本人確認、受取口座登録、規約違反調査中でないことを確認する。
5. 申請日から処理日を決定する。
6. 運営が5日、15日、25日に出金処理を行う。
7. Stripeおよび金融機関の処理結果を出金履歴へ反映する。

## 11. MVP外にする画面

| 画面/機能 | 理由 |
| --- | --- |
| STI検査結果アップロード | 機微情報が重く、MVPでは不要 |
| LINE連携設定 | Web Pushで代替する |
| 正確な地図ピン表示 | 位置推定・待ち伏せリスクが高い |
| リアルタイム緊急ボタン | 運営の即時対応体制がないため |
| 自動出金・即時出金 | 運用確認後に検討 |
| 複数人同時予約 | 安全運用が複雑になるため |
| サブスクリプション | MVPの検証目的から外れる |

## 12. 次工程へのメモ

DB設計では、少なくとも以下の領域を分ける。

* accounts / users / therapist_profiles / user_profiles。
* identity_verifications。
* therapist_menus / pricing_rules。
* locations / service_addresses。
* bookings / booking_status_logs。
* payment_intents / stripe_connected_accounts / payouts。
* messages。
* reviews。
* reports / disputes / refunds。
* admin_audit_logs。

予約ステータス、出金ステータス、本人確認ステータス、タチキャスト審査ステータスは、DB・API・画面で同じ名称を使う。
